## Executive Summary

Epic 2 delivered 8 stories (100%) completing the full query execution and transaction layer for dart-oracledb: SELECT, DML, bind parameters, data type mapping, statement caching, and query error handling — all validated against Oracle 23ai. The epic required a mid-course intervention (Epic 6, Story 6.3) to rebuild the entire EXECUTE wire protocol from scratch after discovering Stories 2.1–2.4 implemented an invented format Oracle silently rejects. The protocol recovery succeeded: 38/38 integration tests pass. Stories 2.5–2.8 shipped on the corrected foundation with genuine integration confidence.

One pre-2.7 critical deferred item (`_maxSizeFor` null VARCHAR bind) was not resolved before Story 2.7 was created, introducing a latent bug in the statement caching layer. This must be addressed before Epic 3 begins.

## What Went Well

### 1. TTC EXECUTE Protocol Recovery (Story 6.3)

Complete rebuild of the EXECUTE wire format against node-oracledb reference, fixing a root cause identical to Story 1.4-FIX. Starting point: 4/38 integration tests passing (ORA-12150 on every call). Ending point: 38/38. The BREAK/RESET MARKER deadlock — Oracle's undocumented DML error protocol — was discovered by inspecting wire traffic and fixed in transport.dart. This prevented the same defect from propagating into Stories 2.5–2.8.

### 2. `sendCommit`/`sendRollback` Timeout (Story 2.5)

A pre-existing correctness gap: a hung server would lock the connection forever. Applied `_receiveDataWithTimeout` to both commit and rollback paths in Story 2.5 (11 patches). This was a pre-2.5 critical from Epic 6 and was correctly resolved before the story closed.

### 3. Negative NUMBER Off-By-One Found in Review (Story 2.6)

Both `decodeNumber` and `encodeNumber` used `102 - digit` instead of `101 - digit` for negative mantissa bytes. The symmetric error hid itself in unit round-trips but failed against real Oracle wire bytes. Caught in review session 3. Production-quality defect, caught before merge.

### 4. Clean Stories 2.7 and 2.8

After the protocol foundation was solid, Statement Caching and Query Error Handling shipped with low friction: 53 unit tests + 6 integration for 2.7, 497 unit tests + 5 integration for 2.8. The inverted test pyramid architecture (from Epic 6) is working.

### 5. CI Pipeline Paying Off (Story 6.4 Investment)

Every Story 2.5–2.8 commit got automatic Oracle 23ai validation. Stories closed with genuine integration confidence rather than "unit tests pass, integration TBD."

### 6. Debug Probes Removed

`tool/dml_probe.dart` and `tool/simple_probe.dart` (hardcoded credentials) were removed from the source tree. The Epic 6 retrospective flagged these as an immediate action item; they are gone.

### 7. deferred-work.md as a Living Document

40+ items captured with context on when they become relevant. Pre-2.7 blockers were correctly identified. Documented debt beats hidden debt.

## Challenges & Root Causes

### 1. Invented Protocol Pattern Recurred (Stories 2.1–2.4)

**Issue:** Stories 2.1–2.4 implemented an EXECUTE wire format Oracle 23ai silently rejects. Unit tests passed because they validated the wrong format against itself. Against a real Oracle server, every EXECUTE closed the connection.

**This is the same defect class as Story 1.4-FIX.** Epic 6 was created specifically to catch it before it propagated. It succeeded — but only because Story 6.3 validated retroactively.

**Root cause:** Stories 2.1–2.4 were written before Epic 6's integration test standards existed. No retroactive gate was applied before those stories were promoted to `dev-complete-pending-validation`.

**Resolution:** Story 6.3 rebuilt the protocol completely. Stories 2.5–2.6 re-validated. All 38 integration tests pass.

**Systemic fix:** "Oracle integration tests passing" must be a hard gate for any story touching protocol implementation — not a post-hoc validation.

### 2. `_maxSizeFor` Null Fix Missed Despite Critical Tag

**Issue:** The Epic 6 retrospective tagged `_maxSizeFor` returning 1 for null VARCHAR binds as a pre-2.7 critical. Story 2.7 was created and completed without the fix.

**Root cause:** The deferred item was in deferred-work.md but not in Story 2.7's acceptance criteria or task list. There was no mechanism to block story creation on unresolved pre-story criticals.

**Impact:** Latent bug in statement caching — null → non-null VARCHAR bind reuse may send wrong `maxSize` to Oracle. Integration tests passed because no test exercises this path.

**Systemic fix:** Any deferred item tagged pre-X.Y-critical must appear as a blocking Task in story X.Y's Definition of Done.

### 3. Code Review Patch Volume Remaining High

Despite improved standards, Stories 2.5 and 2.7 each required 11 patches. Recurring themes:
- Connection state management (re-entrancy, `_isClosed` checks)
- tearDown/setUp cleanup patterns (LateInitializationError, close-before-rethrow)
- Test isolation (shared table PKs, concurrent run collision risk)

These are patterns, not one-offs. They should be in a story template checklist so reviewers have a list to check against rather than discovering them fresh each time.

### 4. Story 2.2 Has No Dedicated Artifact

The result set handling story (OracleResult, OracleRow) was implemented inside Story 2.1 and never captured as a standalone artifact. Documented as a gap in Story 6.3. Still not created.

## Deep Story Analysis Insights

### Common Struggles (3+ stories)

1. **Protocol correctness requiring Oracle validation** (2.1–2.4, 6.3) — Unit tests cannot substitute for real Oracle wire verification. Any story touching TTC/TNS message format must have integration tests against Oracle 23ai as a hard gate.

2. **Review finding substantial issues requiring multiple patch rounds** (2.5, 2.6, 2.7) — Connection state, cleanup, test isolation patterns appear repeatedly. Need a reviewers' checklist for these known categories.

3. **Test isolation with shared Oracle tables** (2.5, 2.6, 2.7) — Fixed PK IDs, shared table names, LateInitializationError in setUp. Parallel test runs against the same Oracle instance can collide. No dedicated test schema/user.

### Breakthrough Moments

1. **BREAK/RESET MARKER Protocol (Story 6.3)** — DML constraint violations require Oracle MARKER/BREAK acknowledgment before the error DATA response arrives. Undocumented. Discovered by watching wire bytes. Fixed the final 20/38 DML integration test failures.

2. **Negative NUMBER Off-By-One (Story 2.6)** — Symmetric encoder/decoder bug hid in round-trips; revealed by Oracle wire. Pattern: always test encode→Oracle→decode round-trips, not just Dart→Dart.

3. **Clean Epic Closure (2.7–2.8)** — After the protocol was solid, the final two stories ran smoothly. Good architecture compounds.

### Velocity Patterns

- **Stories 2.1–2.4 (pre-Epic 6):** "Done" without Oracle validation — invalidated by Story 6.3
- **Story 6.3 (validation sprint):** High-intensity; complete protocol rebuild; resolved 38 test failures
- **Stories 2.5–2.6 (re-validation):** Medium friction; re-validation + code review patches
- **Stories 2.7–2.8 (clean continuation):** Low friction; solid foundation, targeted reviews

## Key Lessons Learned

### 1. `dev-complete-pending-validation` Is the Highest-Risk Status

Every story that carried this label needed a full protocol rebuild when validated against Oracle. Treat it as "broken until proven otherwise." Any story touching TTC/TNS wire format must have Oracle 23ai integration tests passing before it can be marked `done`.

### 2. Oracle Wire Behavior Cannot Be Approximated

BREAK/RESET MARKER, FAST_AUTH, hex-encoded crypto, negative NUMBER encoding — all discovered empirically by running against real Oracle. Unit tests that don't hit a real server validate the wrong thing. The inverted test pyramid is the right architecture for this project, and it is working.

### 3. Critical Deferred Items Need Story-Level Gates, Not Just Documentation

The `_maxSizeFor` miss is the clearest evidence: correctly identified, correctly documented, incorrectly treated as optional at story creation time. Pre-X.Y-critical items must block story X.Y's Definition of Done.

### 4. Code Review Is Catching Real Production Bugs

58+ patches across Epic 2. Negative NUMBER off-by-one, re-entrancy guard missing, rollback-failure swallowing, stale error offset on retry — none of these are noise. The review process is the quality gate.

### 5. Good Infrastructure Compounds

The CI pipeline, inverted test pyramid, and deferred-work.md discipline established in Epic 6 made Stories 2.7–2.8 genuinely smooth. Technical investment in foundations pays off in the stories that follow.
