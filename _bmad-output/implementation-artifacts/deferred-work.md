## Resolved: tolerant TIMESTAMP payload-length decode (2026-06-12)

- `_readTimestampWire` rejected TIMESTAMP payload lengths 8-10/12 (`BufferException` unless exactly 7/11/13 bytes total) while node-oracledb tolerates them — resolved by aligning to node-oracledb `parseOracleDate` parity (product decision, 2026-06-12): the exact-length gate is removed; fractional seconds are read only when total length ≥ 11 (`remaining >= 4`) and TZ bytes only when total ≥ 13 (`remaining >= 6`), with trailing bytes ignored. This cannot desync the stream — the decoder runs on a column-scoped slice from `readBytesWithLength` whose length prefix governs stream position, and the decoders execute only on the strict path (the completion probe consumes via the length prefix and returns before decoding). Sub-7-byte payloads still underflow on the base-byte reads (`BufferUnderflowException`), preserving the probe's "need more packets" signal. `decodeTimestampTz` keeps its separate null-TZ guard, so a sub-13-byte payload reaching the TSTZ decoder still throws `oraProtocolError`. Unit-validated (tolerant 8/9/10/12-byte decode, TSTZ 12-byte rejection, sub-7-byte underflow) and dual-env regression-checked on 23ai + 21c (no live server emits these lengths, so the path is unit-only by design). See [spec-timestamp-tolerant-length.md](spec-timestamp-tolerant-length.md). ## Resolved: probe underflow/malformation distinction (2026-06-12)

- `ttcStreamIsComplete` treated EVERY `BufferException` as "need more packets", including non-underflow malformations (`Integer too large`, sign-bit rejection in `_readInteger`) — resolved: a new `BufferUnderflowException extends BufferException` is thrown only by `ReadBuffer._checkAvailable` (the one true out-of-bytes signal); both completion probes (`ttcStreamIsComplete`, `lobOpStreamIsComplete`) return false only on the underflow subtype and rethrow any other `BufferException` as `OracleException(oraProtocolError)` with the malformation as `cause`; the transport's `isComplete` closure poisons the connection before propagating a probe throw (framing-desync hazard — same pattern as the AC10 cap and mid-query REFUSE). Every other `BufferException` site (sign-bit, integer-too-large, `seek`, TIMESTAMP payload-length drift, OSON offsets/ids) was audited and confirmed a malformation, staying the parent type; real-decoder catches (parent type) are unaffected. Documented limitation (by design, per spec "Never"): a garbage *length* field driving an oversized `skip`/`readBytes` produces the same `_checkAvailable` underflow as a genuine partial stream and intentionally stays "need more packets" under the receive-loop packet cap/timeout backstop — node-oracledb has the identical ambiguity. The exhaustive truncation sweep stays green unchanged. See [spec-probe-malformation-distinction.md](spec-probe-malformation-distinction.md).
