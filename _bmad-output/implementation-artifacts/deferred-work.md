## Deferred from: code review of 2-7-statement-caching (2026-05-21)

- **Concurrent `execute()` on the same connection is undefined** — `OracleConnection.execute()` does not serialize calls; cache hit/miss, `inUse` semantics, and drained `cursorsToClose` all assume serialized callers. Documented contract in node-oracledb thin; user error to overlap futures on one connection. Not Story 2.7 scope.
- **Cached `expectedColumns` becomes stale after DDL changes the SELECT result shape** — `decodeExecuteResponse` decodes new bytes with old column metadata when Oracle omits DESCRIBE_INFO on cached reuse. No invalidation on schema change. Production drivers don't handle this; would require server-side change notifications.
- **Cache key is exact SQL only — bind type changes can mismatch cached cursor's bind metadata** — node-oracledb thin caches per (SQL, bind signature). Needs investigation; ORA-01007 / silent coercion risk if user reuses same SQL with different bind types. Deferred behind Epic 2 error handling.
- **`sendCloseCursors` writes all queued IDs in one message — no SDU/chunking bound** — A long-lived session with `maxSize: 1` could overflow on `closeAll`. Practical risk is low; add bounded chunking later.
- **Cache hit + successful re-execute returning `response.cursorId == 0` triggers `_cache.invalidate(sql)`** — Closes a healthy cached cursor if Oracle ever returns 0 on legitimate re-execute. Integration tests pass; needs protocol verification before changing logic.
- **`SELECT ... FOR UPDATE` is cache-eligible — cursor reuse may change Oracle row-lock semantics across executions.** Needs Oracle-side investigation.
- **`statementCacheSize` has no upper bound — pathological large values (e.g., 2^31) lead to unbounded memory.** Add a documented cap or warning later.
- **Integration test for `statementCacheSize: 1` does not verify cursor reuse / parse-bit clearing** — only that A→B→A doesn't crash. AC3 evidence is weak. Strengthen with `v$open_cursor` query or transport-level instrumentation later.

## Deferred from: code review of 2-6-basic-data-type-mapping patch-resolution pass (2026-05-21)

- **`decodeNumber` no length-based sentinel guard** — If Oracle omits the 0x66 terminator for a negative NUMBER, the decode loop in `decodeNumber` consumes all remaining bytes in the buffer as mantissa digits, desynchronising the packet reader for every subsequent field. `decodeNumber` does not accept a `length` parameter even though `decodeValue` always has one. Pre-existing architectural issue; needs its own story.
- **`encodeNumber` unconditional terminator append for all negative numbers** — `102` is appended after every negative mantissa regardless of length; potential over-long encoding if Oracle's server-side decoder is strict at max field capacity. Validated against Oracle 23ai in practice. Pre-existing.
- **`encodeNumber` throws `FormatException` (not `OracleException`) for `double.infinity` / `double.nan`** — `toStringAsFixed(20)` returns `"Infinity"` / `"NaN"`; `int.parse` inside the digit loop throws an untyped `FormatException`, violating the library's error contract. Pre-existing.
- **`encodeNumber` `toStringAsFixed(20)` floating-point artefacts** — Values like 678.90 produce artefact digits (e.g. `"678.90000000000009..."`); digit extraction then encodes the wrong mantissa. Integration tests use `closeTo` to accommodate. Pre-existing.
- **`decodeNumber` returns `int 0` for `NUMBER(10,2)` zero** — The `0x80` special case returns the int literal `0`, inconsistent with non-zero `NUMBER(10,2)` which returns `double`. Pre-existing; test acknowledges it with a weakened type assertion.
- **Negative mantissa digit-pair=0 not covered by tests** — A negative number containing a zero base-100 digit pair (e.g. -100, -200, -10001) is not exercised in unit or integration tests. Mathematically correct with `101 - 0 = 101` but untested against Oracle wire.
- **`setUp` close() in non-955 else-branch can throw, masking original CREATE TABLE error** — `connection.close()` is called before rethrow; if close() also throws the original exception is replaced. Pre-existing cleanup-in-error-path risk.
- **`tearDown` finally close() can throw, masking non-942 DROP TABLE error** — If DROP TABLE raises a non-942 error and rethrow is reached, the finally close() may itself throw, replacing the more informative DROP error. Pre-existing.
- **Fixed PK ids 8–10 in shared table collide on retry or concurrent runs** — Tests "negative NUMBER int" (id=8), "negative NUMBER decimal" (id=9), and "zero NUMBER" (id=10) use static literal PKs in the shared `test_types_story26` table. Extends the prior-pass defer (ids 1–7). Pre-existing design.
- **DML operations group `tearDown` still uses old `catch (_) {}` pattern** — The Story 2.6 tearDown was improved by this review pass; the adjacent DML operations group was not. Out of scope for this story.

## Deferred from: code review of 2-6-basic-data-type-mapping (2026-05-21)

- **~~🐛 PRODUCTION BUG — `decodeNumber` negative branch returns wrong value~~** — _RESOLVED 2026-05-21 (Story 2.6 dev session 3). Root cause: both `decodeNumber` and `encodeNumber` were applying `102 - digit` to negative mantissa bytes; Oracle's wire format uses `101 - digit` with `102` reserved as the terminator sentinel (confirmed against node-oracledb `parseOracleNumber` at `lib/impl/datahandlers/buffer.js:268`). Symmetric off-by-one in encoder + decoder hid the bug from unit-level round-trip tests but failed against real Oracle wire bytes. Both call sites fixed; both integration tests (`SELECT negative NUMBER returns int`, `SELECT negative NUMBER(10,2) returns double`) now active and passing against Oracle 23ai._
- **TIMESTAMP WITH TIME ZONE / LOCAL TIME ZONE decoder reads only 11 bytes** — `lib/src/protocol/data_types.dart` routes `oraTypeTimestampTz` (181) and `oraTypeTimestampLtz` (231) through the same 11-byte decoder used for plain TIMESTAMP. Wire encoding for TSTZ/TSLTZ carries extra TZ bytes (typically 13 bytes total). End users selecting these column types would get either silent corruption or a buffer error. Pre-existing production code path; needs its own story.
- **DATE/TIMESTAMP round-trip via Dart-bound parameters not exercised** — All temporal INSERTs in the Story 2.6 integration tests use Oracle's `TO_DATE`/`TO_TIMESTAMP` server-side functions with string literals, so the client-side `encodeDate`/`encodeTimestamp` paths are never invoked by the "round-trip" test. Bind-side encoder coverage for temporal types is a separate scope.
- **NUMBER 2^53 int-vs-double boundary not tested at integration level** — `decodeNumber` returns int when `result.abs() <= 2^53`, double otherwise. New tests do not probe near the boundary; a regression that lowered or raised the threshold would not be detected.
- **NUMBER 38-digit / very large values not tested** — Oracle NUMBER supports up to 38 decimal digits; Dart double is ~17. The `_pow100` accumulation loop is not stressed. Document precision-loss contract and add coverage.
- **Bare `NUMBER` (default precision/scale) column not exercised** — Test table only uses `NUMBER(10)` and `NUMBER(10,2)`. A `NUMBER` column without explicit precision is the Oracle default and a likely real-world case.
- **Pure-fraction NUMBER (negative-exponent branch) not exercised by integration** — `decodeNumber` has a dedicated branch for pure fractions (e.g. 0.0001, 0.000001) that decrements the exponent; uncovered by the new tests.
- **TIMESTAMP nanosecond truncation not asserted** — Inputs are µs-aligned (`.123456`); a bug that rounded instead of truncating would pass. Insert a `.123456789` Oracle value via SQL and assert truncation to `.123456`.
- **Pre-epoch / BC and year-boundary DATE values not tested** — Year-1, year-100, year-9999, BC dates. Encode/decode century-byte arithmetic asymmetry is plausible but uncovered.
- **CHAR(5) padding semantics not validated** — Story 2.6 test inserts exactly 5 chars (`'ABCDE'`), avoiding Oracle's right-pad behavior. The interesting contract question (does the driver return the padded form or trim?) is sidestepped.
- **Multi-byte UTF-8 in VARCHAR2/VARCHAR/CHAR not tested** — `utf8.encode`/`utf8.decode` paths are correct in principle but the integration suite never exercises non-ASCII (emoji, CJK, accents). NLS_LENGTH_SEMANTICS interactions also untested.
- **Empty-string vs NULL conflation in VARCHAR2** — Oracle treats `''` as NULL for VARCHAR2 — a well-known gotcha. The Story 2.6 NULL test asserts only the explicit-omit path; the implicit-empty-string path is not differentiated.
- **VARCHAR2(100) length boundary and over-length (ORA-12899) not tested.**
- **Shared fixed table name `test_types_story26` and fixed PK ids 1..7** — Parallel runs against the same DB collide; setUp 955→TRUNCATE fallback can also race with a concurrent DROP. Test isolation strategy is a cross-cutting concern with the other integration test groups in this suite.
- **No connect timeout for `OracleConnection.connect` in setUp** — A hung listener stalls the suite until `dart test`'s global timeout. Pre-existing pattern in integration suite.
- **`hasOracle` env-var presence vs value** — `Platform.environment.containsKey('RUN_INTEGRATION_TESTS')` evaluates true for any value (including `0` or empty). Already deferred from 6-4 review; normalize once across the suite.
- **`oraTypeInteger`/`oraTypeFloat`/`oraTypeVarnum` switch arms + absent BINARY_FLOAT/BINARY_DOUBLE/LONG/CLOB/NCHAR/NVARCHAR2** — `decodeValue` covers some legacy OCI type codes but they are not exercised by the new tests; BINARY_FLOAT/BINARY_DOUBLE/CLOB fall to the "Unsupported Oracle type" default and may surface in real schemas. Separate type-coverage scope.

## Deferred from: code review of 2-5-transaction-management (2026-05-21)

- **AC8 has no end-to-end test that forces the timeout path** — `_receiveDataWithTimeout` is exercised only via the default 30-second value; no test ever blocks a server response to prove the timeout actually fires with a useful `OracleException`. Spec Task 2.3 allowed documenting why ("no mock hook in transport tests; fake server out of scope") and that documentation exists in the Dev Agent Record. Add a mock-socket transport unit test when the transport test infra gains a hook, or an integration-style test using a pausable network proxy.
- **Transport sequence-desync after RPC timeout** — `_receiveDataWithTimeout` (transport.dart:386-388) already documents: "on timeout the underlying socket read continues; subsequent operations may receive stale data." This is a known hazard for all timeout-bounded RPCs, not just commit/rollback. Mark the transport as broken (force disconnect or set `_corrupted = true` checked by `_ensureOpen`) on timeout so the next RPC fails loudly rather than reading the orphaned response.
- **`_ensureOpen()` does not detect silently-dropped sockets** — Only checks `_isClosed`. TCP RST or idle-firewall-kill is only observed after the 30s RPC timeout fires. Cross-check `_transport.isConnected` (or its socket-level equivalent) inside `_ensureOpen` so dead-connection RPC attempts fail fast.

## Deferred from: code review of classical-auth-fallback (2026-05-21)

- **`sendAuthPhaseOne` lacks a timeout wrapper** — `AuthFlow.authenticate` wraps AUTH_PHASE_TWO's `receiveData` in `authTimeout` to handle Oracle's silent-close on wrong password, but the classical AUTH_PHASE_ONE has no timeout. A pre-23 server that silently closes after sending an internal error (dead instance, listener misroute) would hang indefinitely. The 21c wrong-password integration test currently passes because 21c errors out at phase two, not phase one. Add a timeout if a real hang is ever observed.
- **Classical integration test imports `package:oracledb/src/transport/...`** — `classical_auth_integration_test.dart` cracks open the `src/` tree to probe `Transport.supportsFastAuth` before the test starts. Other integration tests do similar. Acceptable for now; consider exposing a test-only API (or a `dart_oracledb_internal` library) if internal layout changes start churning tests.
- **`updateState(phaseOneSent)` runs before the classical branch sends anything** — If `sendProtocolNegotiation` throws inside the classical branch, the AuthFlow state shows `phaseOneSent` despite no phase-one packet on the wire. State-machine inspectors (retry/recovery logic) would be slightly misled. Low impact; move the `updateState` call into each branch if state observability becomes important.
- **`RUN_INTEGRATION_TESTS` skip logic checks presence, not value** — `Platform.environment.containsKey('RUN_INTEGRATION_TESTS')` runs the tests even when the value is `false`. This matches the inconsistent pattern already deferred from 6-4; normalize once across the integration suite.
- **`use23aiFormat` for AUTH_PHASE_TWO is gated on `_ttcFieldVersion`, not `supportsFastAuth`** — In `auth.dart:411` `transport.shouldWriteTokenNumber` depends on `_ttcFieldVersion` set from `compileCaps`. If a pre-23 server returns a short `ProtocolResponse` with null `compileCaps`, `_ttcFieldVersion` stays at its default (`24`) and `use23aiFormat=true` is sent against a pre-23 server. Empirically 19c/21c return compile caps so this is defensive. Consider gating `use23aiFormat` on `transport.supportsFastAuth` instead — more robust to short responses.
- **`AuthPhaseOneResponse.decode` swallows ERROR TTC messages** — Pre-existing: on `msgType == ttcMsgTypeError` the decoder logs a warning and returns a default response with empty `sessionData`, leading to a downstream phase-two failure with garbage inputs instead of a fast surface error. Was already buggy for FAST_AUTH; the new classical path inherits the same behavior. Fix by raising an `OracleException` with the parsed Oracle error code inside `AuthPhaseOneResponse.decode`.
