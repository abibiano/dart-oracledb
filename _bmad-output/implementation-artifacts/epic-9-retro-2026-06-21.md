Key shipped capabilities:

- `OracleBind.out(type: OracleDbType.cursor)` as the PL/SQL `SYS_REFCURSOR` OUT bind type.
- Cursor OUT bind decodes into `OracleResultSet` with metadata-before-fetch, `getRow()`, `getRows()`, early close, pool release, and the existing concurrent-operation guard.
- Cursor-valued SELECT columns (`CURSOR()` subqueries) eagerly materialize into `List<OracleRow>` on all three result paths: eager execute, `queryStream()`/`executeStream()`, and `execute(resultSet: true)`.
- `OracleResult.implicitResults` (node-oracledb parity name) carrying either `List<OracleRow>` (eager) or `OracleResultSet` (lazy) per implicit result cursor.
- Two-slot multi-cursor ownership: `_openResultSet` for the single REF CURSOR/SELECT handle, `_openImplicitResultSets` for lazy implicit result groups; connection is owned until every handle closes.
- Eager implicit-result safety-cap exhaustion fails loud (no silent truncation).
- Nested cursors inside implicit results fail loud consistently on both Oracle environments (pre-23/21c protocol limitation documented and deferred).

## What Went Well

### The Epic 8 Engine Held

The biggest architectural win is negative: nothing had to be redesigned. `OracleResultSet`, `ResultSetCursor`, `releaseResultSet()`, `forceCloseOpenResultSet()`, the close-cursor piggyback queue, and the `_executeInProgress` round-trip guard all composed correctly with the Epic 9 cursor shapes. Epic 8's investment in a general lazy cursor lifecycle paid immediately.

### Two-Slot Multi-Cursor Ownership Was Clean

Story 9.3's multi-cursor ownership design (`_openResultSet` + `_openImplicitResultSets`) is minimal and legible. Mixed ownership — one REF CURSOR OUT bind plus one or more lazy implicit results — releases correctly when every handle closes. Story 9.4's coexistence integration test proved this against both Oracle environments.

### Oracle 21c Bug Found and Fixed in Scope

Story 9.2 found a real pre-existing bug in the SELECT column decode path: `_decodeColumnValue` did not thread the negotiated `ttcFieldVersion` into embedded cursor describe parsing. The 23ai version happened to match the defaulted version; 21c did not, producing ORA-12547. The fix was one line plus a focused regression test pinning the pre-23 (field version 20.1) decode shape. This is exactly the value of dual-env testing.

### User-Approved Scope Decision on AC3

When Story 9.4's dual-env testing surfaced that nested cursors inside implicit results cannot work on Oracle 21c without out-of-scope per-cursor describe round-trips, the decision to defer and pin fail-loud instead was made quickly with documented byte-level evidence. The connection stays reusable; the behavior is consistent across both environments; the deferral is clean.

### Adversarial Review Caught Real Defects Every Story

- 9.1: Multi-cursor cursor-id cleanup before throwing; zero-column and unsupported-column descriptor rejection.
- 9.2: Parent cursor not queued when nested materialization fails before result-set handoff; duplicate cursor ids from distinct `DecodedCursorResult` objects.
- 9.3: Eager drain silent truncation made fail-loud; dead code AC7 handler moved to its live site; `close()` not clearing `_openImplicitResultSets`.
- 9.4: AC3 scope wording aligned; lazy ownership documented as close-only (matching node-oracledb); pool test cleanup guard; AC5 cache evidence added.

Fourteen production-code patches across the four reviews, every one catching a real defect or documentation gap.

### Reference-First Approach Held

Every wire shape was verified against the local `reference/node-oracledb` copy before implementation. Story 9.4 captured raw 21c wire bytes directly using temporary Transport instrumentation before making the deferral decision. No guesswork; evidence first.

## Challenges

### The 21c `ttcFieldVersion` Bug Class Is Persistent

Story 9.2 is the second time a cursor-describe path defaulted `ttcFieldVersion` to 24 instead of using the negotiated version. Story 9.2's Dev Notes documented this as a "bug class" warning for all future embedded-describe paths. Story 9.3 explicitly called it out and threaded the version correctly. But the root cause — having to manually thread a negotiated version through decode call chains — means this will keep surfacing whenever a new cursor decode path is added.

Winston (Architect): "The underlying problem is that `ttcFieldVersion` lives on the transport negotiation and has to be threaded manually into every embedded decode call. A refactor to make it ambient or injectable would prevent the class permanently."

### Nested Cursors Inside Implicit Results Hit a Protocol Wall

The AC3 deferral was clean, but it represented a real discovery: Oracle 21c does not transmit nested cursor describe metadata inline in implicit results. The 23ai patch worked; the 21c path requires a separate per-cursor describe/execute round-trip per nested cursor id — a new protocol capability, not a tweak. This is a genuine capability gap that matters for applications using nested cursors in PL/SQL result sets and supporting databases below version 23.

### Eager Implicit-Result Silent Truncation Was a Latent Defect

Story 9.3 implemented the eager drain safety backstop (`maxFetchIterations × prefetchRows ≈ 50k rows`) without a fail-loud signal on breach — the implicit result would silently return a truncated `List<OracleRow>`. The code review caught this and the decision was made to fail loud. This is the third time a silent-cap/silent-truncation issue has been caught during review (Stories 7.9 and 9.3). The pattern suggests there is a systemic gap in the upstream requirement to always fail loud when bounds are exceeded.

### Multi-Story Arc Accumulated Review Debt

The four stories were implemented sequentially, each building on the last. By Story 9.4, the review was finding minor wording issues, comment staleness, and small test coverage gaps rather than major defects. This is healthy, but it means the review load is not uniformly distributed — Stories 9.1 and 9.3 had the highest defect density. A more aggressive review at the design-decision level before each story starts could shift some of that load forward.

Dana (QA Engineer): "Nine-point-four was mostly a test story, but it still generated five patches. Four were presentation/wording; one was a real pool-test cleanup gap. For a 'primarily test work' story, that ratio is fine."

## Key Insights

1. **A shared cursor engine is only as valuable as its lifecycle semantics are proven.** Epic 8 proved the lifecycle. Epic 9 got to reuse it. The investment in Epic 8's adversarial lifecycle review paid across four more stories.
2. **`ttcFieldVersion` must be ambient or injected — manual threading will keep producing the same bug class.** The Story 9.2 fix is correct but fragile; a future refactor should eliminate the manual threading obligation.
3. **Dual-environment testing found a real production defect in Story 9.2 that 23ai-only testing would have missed entirely.** This is the fourth straight epic where at least one genuine bug was found only on 21c.
4. **The fail-loud rule is worth repeating: any safety cap that silently truncates is a defect.** Story 9.3 caught this for eager implicit result draining; it has now appeared three times in the project.
5. **The two-slot ownership model is clean but has a known limitation: it does not support multiple concurrent REF CURSOR OUT binds.** That constraint is documented, intentional for this epic, and deferred; but it will need to be resolved before any PL/SQL procedure that returns two OUT cursor binds can be supported.
6. **When a dual-env integration test surfaces a fundamental pre-23 protocol capability gap, the right response is byte-level evidence, a clean deferral, and consistent fail-loud — not a version-gated patch.**
