## Challenges

### 1. The Invented Protocol Pattern Recurred

**Issue:** Stories 2.1–2.4 implemented an EXECUTE wire format that Oracle 23ai rejects. Unit tests passed because they validated the same invented format. Against a real Oracle server, every EXECUTE was silently rejected with connection close.

**This is the same defect class as Story 1.4.** Epic 6 was created specifically to prevent this from propagating to Stories 2.5+.

**Root cause:** The original Epic 2 stories were written before Epic 6's test standards existed, and no mechanism existed to retroactively require integration validation before those stories were promoted.

**Impact:** Stories 2.5 and 2.6 have been downgraded from review/done to `dev-complete-pending-validation`.

### 2. BREAK/RESET MARKER Protocol (Hidden Oracle Behavior)

**Issue:** Oracle sends MARKER/BREAK packets before the error DATA response for DML constraint violations. The client must send a RESET MARKER acknowledgment before Oracle will send the actual error. Without this, the client waits for DATA that never arrives — a silent deadlock (30-second timeout in integration tests).

**Resolution:** `_receiveAllTtcData()` now detects BREAK packets and sends `_sendResetMarker()` before continuing to read. This fixed the final 20/38 integration test failures.

**Why this matters:** This behavior is not documented in Oracle's public docs. It was discovered by inspecting actual wire traffic against Oracle 23ai.

### 3. Deferred Items Accumulation

40+ deferred items documented across Stories 6.2, 6.3, and 6.4. Most are low-priority, but two are landmines for specific future stories:

- `_maxSizeFor` returning 1 for null-valued VARCHAR bind → becomes a bug when Statement Caching (Story 2.7) is added
- `sendCommit()`/`sendRollback()` have no timeout → relevant to Transaction Management (Story 2.5)

### 4. Debug Probe Tools in Source Tree

`tool/dml_probe.dart` and `tool/simple_probe.dart` with hardcoded credentials (`system/testpassword`) were committed as one-off debugging artifacts during Story 6.3. They must be removed before any new story begins.

## Key Insights

1. **The `dev-complete-pending-validation` status is a time bomb.** Every story carrying this label has a real risk of implementing a protocol Oracle rejects. Story 6.3 validated this 100% of the time on Stories 2.1–2.4. The label needs to be treated as "potentially broken."

2. **Protocol correctness requires Oracle 23ai. Full stop.** No unit test arrangement can substitute for running against the actual Oracle server. The inverted test pyramid is the right architecture for this project.

3. **node-oracledb reference consultation must be a gate, not a suggestion.** It is available in `reference/node-oracledb/lib/thin`. Before implementing any TTC/TNS wire format, compare with the reference. This must be an explicit task in every protocol story, not an ad-hoc rescue.

4. **CI/CD changes the safety net from optional to automatic.** Story 6.4 means Oracle validation is no longer dependent on a developer remembering to run Docker. This permanently raises the floor.

5. **Documented debt beats hidden debt.** 40+ deferred items is a significant number, but they are all explicitly tracked in deferred-work.md with context on when they become relevant. This is much better than the alternative.

## Significant Discoveries — Epic Update Required

### ⚠️ Stories 2.5 and 2.6 require re-validation before Epic 2 can continue

**Discovery:** The TTC EXECUTE protocol was rebuilt from scratch in Story 6.3. Stories 2.5 and 2.6 were implemented and reviewed against the old invented format. Their integration tests are therefore invalid.

**Before Story 6.3, these stories assumed:**
- `sendExecute()` signature is stable
- Transport response handling is correct
- Commit/rollback flow works end-to-end

**Story 6.3 revealed:**
- `sendExecute()` signature changed (added `isQuery` parameter, `ttcFieldVersion`, `sequence`)
- `_receiveAllTtcData()` now handles BREAK/RESET MARKER packets
- Transport response processing was materially rewritten

**Recommendation:** Re-validate Stories 2.5 and 2.6 as the first act of Epic 2 continuation. A single validation story (similar in scope to Story 6.3, but smaller) covering both stories is the cleanest approach.

**Do NOT start Story 2.7 until 2.5 and 2.6 re-validation is complete.**
