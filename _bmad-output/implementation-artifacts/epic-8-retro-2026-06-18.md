## What Went Well

### Shared Engine Decision

The biggest win was unifying eager and lazy fetching around `ResultSetCursor`. That avoided a split-brain implementation where `execute()`, `OracleResultSet`, and streams decode rows differently. The cost was high regression pressure, but the story records show that pressure was handled with targeted unit tests and full dual-env integration runs.

### Public API Stayed Stable

Story 8.3 avoided a weak `dynamic` or generic `execute()` return shape. `execute()` still returns `OracleResult`; result-set mode populates `OracleResult.resultSet` and keeps eager `rows` behavior unchanged by default. That is a strong Dart API outcome.

### Lifecycle Discipline Was Proven, Not Assumed

Story 8.4 validated the hard parts developers often skip: explicit subscription cancellation, `await for` break cleanup, natural stream completion, overlapping pulls, pool release of open result sets, and public result-set pool cleanup.

### Statement Cache Safety Became Testable

Story 8.5 converted cache safety from an architectural claim into executable coverage. It proved cache pinning, natural-drain reuse, failure invalidation, bind-signature identity, in-use cache transitions, and disabled-cache cleanup. The disabled-cache cursor leak was found by a focused test and fixed without API churn.

### Incidental Defects Were Captured Cleanly

The `ping()` TTC-stream desync was out of scope for Story 8.5, but it was isolated, documented in `deferred-work.md`, and not mixed into the story fix. That is healthy scope control.

## Challenges

### The Eager Path Rewrite Was High Blast Radius

Story 8.1 had to rewrite eager `execute()` onto the lazy engine. That was necessary, but it made the story riskier than a normal public API addition. The review correctly found high-severity error-path defects around streaming FETCH failures and per-batch LOB materialization failures.

### API Shape Needed Reconciliation

Story 8.3 initially carried tension between a named `options:` expectation and Dart's existing optional positional bind signature. The final positional third argument is reference-faithful and source-compatible, but the story text had to be corrected after review.

### Several Cursor Edge Cases Remain Deferred

Open items remain around describe-mismatch retry returning cursor id 0, failed result-set release invalidating by key instead of the held entry, failed result set plus connection close cleanup, `StatementCache.release()` object identity, `fetchSize=1`, and the duplicate-column cross-batch sentinel. None blocks Epic 8 completion, but several are directly relevant to Epic 9.

### Planning Still Under-Specifies Multi-Cursor Ownership

Epic 9 will add REF CURSOR OUT binds, nested cursors, and implicit results. Architecture says these decode into `OracleResultSet`, but Story 8's single-open lazy handle discipline creates design questions when one `execute()` may return multiple cursor handles.

Winston (Architect): "Epic 9 cannot just say 'return OracleResultSet'. It needs an explicit ownership model for multiple server-backed cursors returned by one statement."

Dana (QA Engineer): "And every one of those handles needs close, cancellation, failure, and pool-release behavior pinned in tests."

---

## Key Insights

1. A lazy database cursor is not just a collection API. It is a connection ownership model.
2. Statement-cache safety depends on object lifetime, not just SQL text and bind shape.
3. Public API parity decisions need reference evidence before implementation, especially where Dart syntax differs from JavaScript.
4. Dual-env validation remains the strongest signal for this driver because Oracle 23ai and Oracle 21c exercise materially different auth paths.
5. Out-of-scope defects should be isolated and logged immediately, not quietly absorbed into unrelated stories.
6. Epic 9 needs story-level lifecycle ACs, not just feature ACs.

## Next Epic Preview: Epic 9 - REF CURSOR & Implicit Results

Epic 9 depends directly on Epic 8. Planning and architecture define the direction:

- REF CURSOR OUT binds should decode into `OracleResultSet`.
- Nested cursors should use the same lazy result-set model.
- Implicit results from `DBMS_SQL.RETURN_RESULT` require `OracleResult` to expose multiple result sets.
- The existing `ttcExecOptionImplicitResultset` flag is already set, but results are not consumed yet.
- `ExecuteResponse` currently carries a single metadata/row set, so multiple result-set handling needs design and tests.

Preparation concerns:

- Define whether multiple returned cursors can be open concurrently on one connection, or whether consumption must be serialized.
- Specify how connection ownership is held while implicit result sets remain unclosed.
- Decide close behavior when one cursor fails while sibling cursors still exist.
- Ensure REF CURSOR OUT binds use the same cache, close, pool, and failure-invalidating discipline proven in Epic 8.
- Add explicit tests for no silent consumption of unknown cursor types.

Significant discovery assessment:

- No fundamental change to Epic 9's direction is required.
- Epic 9 story specs should be updated before development to include explicit multi-cursor ownership, cleanup, and failure semantics.
