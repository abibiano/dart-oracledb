# Development Guide — oracledb (dart-oracledb)

> Generated by the Document Project workflow (deep scan), 2026-06-12. The
> authoritative human walkthrough is `CONTRIBUTING.md`; this is the condensed,
> agent-oriented version. The Oracle-runtime nuances on Apple Silicon also live
> in the repo's `CLAUDE.md`.

## Prerequisites

- **Dart SDK `^3.12.0`** (pure Dart; no Oracle Client / Instant Client needed).
- **Docker** for the integration databases. On **Apple Silicon** you also need
  **Colima + QEMU** for the emulated 21c target:
  `brew install colima qemu lima-additional-guestagents`.

## Install & analyze

```bash
dart pub get
dart analyze            # MUST be zero warnings (strict-casts/inference/raw-types)
dart format .           # prefer_single_quotes, prefer_const, prefer_final_locals
```

`analysis_options.yaml` enables strict analysis via `package:lints`. Use
`package:logging`, never `print` (`avoid_print`).

## Build

It's a library — there is no build step. Consumers depend on it via `pubspec`.
Validate it compiles/analyzes cleanly and that `dart doc` renders.

## Running tests

### Unit tests (no database)

```bash
dart test test/src/
```

Unit tests mirror `lib/src/` and cover buffers, codecs, message encode/decode,
crypto, the SQL classifier, and the statement cache.

### Integration tests (real Oracle, BOTH versions required)

This project validates against **two** Oracle versions, and on Apple Silicon each
runs in a **different** container runtime:

| Version | Runtime / docker context | Host endpoint |
|---------|--------------------------|---------------|
| **Oracle 23ai** (primary) | Docker Desktop — `docker context use desktop-linux` | `localhost:1521` / `FREEPDB1` |
| **Oracle 21c** (secondary, required) | Colima x86_64 VM — `docker context use colima` | `localhost:1522` / `XEPDB1` |

**Rule of thumb: Docker Desktop for 23ai, Colima for pre-23 (21c).** Never run
21c under Docker Desktop's emulation — it crashes Oracle init.

```bash
# Bring up 23ai (native ARM64)
docker context use desktop-linux
docker compose up -d

# Bring up 21c (emulated x86_64) — one-time: colima start --arch x86_64 --cpu 6 --memory 8
docker context use colima
docker compose --profile oracle21c up -d oracle21c     # name the service!

# Run against 23ai (FAST_AUTH path)
RUN_INTEGRATION_TESTS=true dart test test/integration/

# Run against 21c (classical AUTH_PHASE_ONE/TWO path)
RUN_INTEGRATION_TESTS=true ORACLE_PORT=1522 ORACLE_SERVICE=XEPDB1 dart test test/integration/
```

**Gotchas (from `CLAUDE.md`):**

- Always **name the `oracle21c` service** in `up`; a bare `--profile oracle21c up`
  also starts the profile-less 23ai service *inside Colima*, starving the VM.
- 2 CPU / 4 GB crashes 21c init (`ORA-04021`) — use **6 CPU / 8 GB**.
- `docker context` only switches the daemon the CLI talks to; both DBs can be up
  at once on different ports, so a single run can target 1521 and 1522.
- Colima does not autostart; after a macOS reboot run `colima start` before 21c.
- 21c first-boot init takes a few minutes under emulation — wait for
  `health: healthy` (`docker compose ps`) before connecting.

## ⚠️ Dual-environment validation is mandatory

**Every feature and bug fix must add integration tests that pass on BOTH Oracle
23ai and Oracle 21c, and must be validated on both before it is "done."** Running
only 23ai is never sufficient — the classical auth path (pre-23) is a first-class
supported target.

When writing integration tests:

- **Never hardcode** `FREEPDB1`, `XEPDB1`, `1521`, `1522`, or credentials. Use the
  `test/integration/test_helper.dart` getters (`testHost`, `testPort`,
  `testService`, `testUser`, `testPassword`, `testConnectString`).
- Auth-path-specific tests (e.g. `classical_auth_integration_test.dart`) must use a
  `Transport.supportsFastAuth` probe to auto-skip on the wrong server.
- All other integration tests should pass on **both** versions without skipping.

## Coverage

```bash
dart test --coverage=coverage
dart pub global run coverage:format_coverage \
  --lcov --in=coverage --out=coverage/lcov.info --report-on=lib
```

A coverage floor is enforced in CI (see `.github/workflows/`). The historically
measured baseline is ~68–70%; the enforced floor reflects that, not the default 80%.

## CI

`.github/workflows/` runs analyzer + unit tests, plus two real-DB jobs —
`integration` (23ai) and `integration-21c` — with readiness probes and
docker-log diagnostics. Mirror the dual-env discipline locally before pushing.

## Adding a protocol feature (the usual loop)

1. Find the wire format in `reference/` (node-oracledb thin driver) — it is the
   canonical answer for byte layout.
2. Add/extend the codec in `lib/src/protocol/data_types.dart` (or `oson.dart` for
   JSON), using **explicit-endian** buffer methods only.
3. Add/extend the message in `lib/src/protocol/messages/` (most paths touch
   `execute_message.dart`).
4. Wire it through `lib/src/transport/transport.dart` and surface it on
   `OracleConnection` / the public value types if user-facing.
5. Unit-test the codec/message in `test/src/...`; add an integration test in
   `test/integration/...` that passes on **both** Oracle versions.
6. `dart analyze` clean, `dart format`, then run both integration suites.

## Code style essentials

- Relative imports within `lib/src/`; `package:oracledb/` only in tests/examples.
  Import order: `dart:` → `package:` → relative.
- Always `await` futures immediately or `unawaited(...)`; close all sinks/subscriptions.
- Preserve error context: `throw OracleException(..., cause: original)`.
- Every resource gets an explicit `close()` and a try/finally (or `withConnection`).
- **Security:** never log or store passwords — only derived keys. Prefer bind
  parameters over string interpolation.
- **Crypto:** `pointycastle` only; do not add `package:crypto`.

## Submitting changes

See `CONTRIBUTING.md` for the full PR process. Before opening a PR: analyzer
clean, unit + both integration suites green, and any 21c-specific deviation either
fixed or explicitly documented as a guarded skip.
